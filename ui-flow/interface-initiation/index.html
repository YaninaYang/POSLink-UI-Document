    Interface Initiation

    • Invoking the UI App will always be done by a BroadPOS App. BroadPOS will not accept any input from the UI App before this initiation. So, the UI App will not need to be running to be activated.

    • Initiation is always done by Activity Intents. And every interface needed to continue the flow of a transaction has a unique identifier (Intent Action).

    • BroadPOS does not expect the UI App to implement all the interfaces it may need to complete a transaction. If the UI App does not implement an interface, BroadPOS will bring to foreground its default interface. This modularity of breaking down a transaction into individual interfaces relieves the UI App of the responsibility to create every single interface a BroadPOS App needs to show.

    • If the UI App intends to implement an interface, it needs to put the unique identifier (Intent Action) in the intent-filter of an activity in the Manifest. For example, the “Enter Amount” interface is used to collect the base amount to be paid in a transaction. The unique action that identifies this interface is “com.pax.us.pay.action.ENTER_AMOUNT”. The BroadPOS App will call startActivity() with an intent that has been instantiated with the action . One thing to keep in mind, as of Android 12, android:exported must be set, to make the activity available to other apps such as BroadPOS.

    <activity android:name = "SampleUIAppActivity" android:exported = "true">
    <intent-filter>
    	<category android:name="com.pax.us.pay.ui.category.TEXT" />
   	<category android:name="android.intent.category.DEFAULT" />

    	<action android:name="com.pax.us.pay.action.ENTER_AMOUNT" />
    </intent-filter>
</activity>
    • We are calling such actions, Interface Initiation Intent Action. And the corresponding intet comes bundled with information the UI App needs to use as inputs. These are the Interface Initiation Input Extras. They are action specific. However, some input extras are sent with all intents. It frees the UI App from storing states or any knowledge of the ongoing POS transaction the interface is a part of. For example, the package name of the BroadPOS Application the UI App will need to submit back to, or the type of the ongoing POS transaction (Debit Sale, Credit Return, etc.). All the input extras are listed along with the list of actions here.
