    Response Broadcasts
    • EntryResponse.ACTION_ACCEPTED
    • EntryResponse.ACTION_DECLINED

    The BroadPOS App responds to the Request Broadcasts used to submit output of an interface, identified by the action EntryRequest.ACTION_NEXT. As we know, there are two more Request Broadcasts: suffixed with ABORT, and SECURITY_AREA. These do not generate any responses. In response to EntryRequest.ACTION_NEXT, there can be two types of Response Broadcasts:

    EntryResponse.ACTION_ACCEPTED #

    • This broadcast does not contain any extra data bundled with it.
    • This is the conclusion of an interface identified by a unique Interface Initiation Action name. No more submissions (Request Broadcasts) from this interface will be received after this.
    • BroadPOS will begin the Interface Initiation stage of the next Interface after this. If the next Interface Initiation Action is declared in the UI App Manifest, the UI App can expect another request there. If the next action is also caught by the same Activity, and the android:aunchMode is unmodified, a new Activity will be created.
    • However, if this was the final interface of the ongoing transaction, no new Interface Initiation will take place. The BroadPOS App will broadcast the Disconnect Message instead.

    EntryResponse.ACTION_DECLINED #

    • Declined Broadcasts are sent from the BrodPOS App in case of invalid input or unsupported requests. It is bundled with a response code and a message. Intent Extras EntryResponse.PARAM_CODE: long Non-null

    EntryResponse.PARAM_MSG: String Non-null

    • Unlike the Accepted response, BroadPOS does not take any further steps after sending the Declined Broadcast. It is consumed by the UI App and BroadPOS expects another Request Broadcast (Retry) from that interface. There are cases, when instead of waiting, the BroadPOS App re-initiates the interface by calling that Activity again. For sensitive interfaces, BroadPOS re-initiates the interface with changed bundled data that indicates the finiteness of trial and error. For example, for the SecurityEntry.ACTION_ENTER_PIN interface, there is a bundled extra that shows if it is the first (“NORMAL”), second (“RETRY”), or final (“LAST”) try.
    • The bundled extras can be used to notify the end user about the situation. The possible response extras are listed below.
